How to write an ACM programming contest problem

Writing a programming contest problem is not terribly hard, but does take time and attention to detail. This document is meant to give you guidance on producing a successful problem.

Come up with an idea. A good contest problem is one that

  • Is challenging to some fraction of the target audience. We usually have at each contest a couple of easy problems suitable for beginning programmers, 3-5 medium difficulty problems designed for students who have taken a data-structures course, and a couple of harder problems designed to give advanced students something meaty to work on.

  • Has an easy description. Problem's whose descriptions fit on a single page are best. If it's hard to describe a problem, then it's probably even harder for the participants to understand it.

  • Can be effectively tested. Contestant submissions are tested for correctness by running them multiple times on various test case inputs and comparing program outputs with known correct results. The "known correct results" must be known correct with very high confidence and the test cases should be designed to flush out not-completely-correct submissions.

  • Is interesting to think about and solve, and does not require a lot of tedious coding. We want contestants to enjoy the contest, so we try to couch the problems in interesting scenarios, make the problems intrinsically attractive, and not require gobs of code to solve.

  • Doesn't have a trivial solution in any of the allowed programming languages (C, C++, Java, Python). For example, "given a 20 digit number, determine if it is prime" would be a pretty good challenge for C/C++, but would not be a good challenge for Java because it could be solved in a few lines of code using BigInteger.isProabablePrime.

  • Requires a program that processes data non-interactively. Programs should read data from Standard Input and write data-dependent results to Standard Output, and require no interaction with a user.

  • Is floating-point rounding-error friendly because it either does not involve floating-point (preferred) or it's straightforward to produce answers that are within some tolerance of correct (eg, "results within 0.001 of correct will be accepted").

Write a description. The contest documents are produced using LaTeX, which takes a text file as input and produces a PDF output. The text allows tagging any part with formatting tags to achieve italic and bold text, and can import PDF, PNG and JPEG graphics which have been tightly cropped to their image. The preferred format looks like this:

\problemname{A--maze--ing!}

We are going to generate a maze from a 5X5 grid of rooms. Every room has four walls except for walls we specify as missing. The top left and bottom right rooms have a missing wall as shown in the first diagram below. Additional walls will be specified as missing by giving you a pair of adjacent room numbers, indicating that the wall separating them is missing. The rooms are numbered as shown in the second diagram below. The third diagram below is the maze that results from the sample input given at the bottom of the page.

{\small\begin{verbatim}
   _ _ _ _                              _ _ _ _ 
|_|_|_|_|_|     | 1| 2| 3| 4| 5|     | |_  |_|_|
|_|_|_|_|_|     | 6| 7| 8| 9|10|     | |_| |_ _|
|_|_|_|_|_|     |11|12|13|14|15|     |_ _  |_|_|
|_|_|_|_|_|     |16|17|18|19|20|     |_|_|_  |_|
|_|_|_|_| |     |21|22|23|24|25|     |_ _|_|_  |
 \end{verbatim}}

\section*{Input}

Each line of input contains a pair of integers, indicating a pair of rooms whose shared wall is missing. A line with a pair of zeros indicates the end of the input.

\section*{Output}

Your program should output any sequence of rooms that leads from Room 1 to Room 25 without going through any existing wall or visiting any room more than once. Such a path is guaranteed to exist. Only horizontal and vertical movement is allowed.

\includesample{sample}

The problemname tag specifies what the title of the problem is and the title appears at the top of the typeset page. The includesample tag reads sample.in and sample.ans and typesets them side-by-side in a box at the bottom of the page to give the contestant a simple example of what the input will look like and what the output should look like.

Left quotes should be made with one or two backticks (` for single and `` for double) while right quotes should be make with one or two single-quotes (' for single and '' for double). To make some text italic, bold, or monospace (ie, courier font): \textit{this will be italic}, \textbf{this will be bold}, and \texttt{this will be monospace}. If you're comfortable doing so, you may use LaTeX math notation. As seen in the example, if you want several lines to be in a monospace font, bracket the lines with verbatim. If you want to include a graphic: put something like \includegraphics[scale=0.4]{LightsOutIllustration.jpg} where you want the graphic to appear and to what scale you want. The problem set editor will fine tune all placements and scales.

Share your writeup with other people who have experience in writing or solving programming contest problems to get feedback on clarity and difficulty. Revise your writeup based on this feedback.

Write a sample solution. You should write a sample solution in C, C++, Java or Python. This is the best way to convince yourself that the problem really is at the difficulty level you intend. Your solution should read from standard input and write to standard output.

Write test data. You should write a simple and short sample.in and sample.ans. These will go into the problem statement and should give contestants just enough information to clarify the problem statement, but should not elaborate all the edge and corner cases that they should think of themselves. Then write as many in and ans pairs needed to establish that any program is likely correct. This means including edge, corner and larger cases that are likely to give a wrong or incomplete solution trouble. I normally give my test inputs the names 1.in, 2.in, 3.in, etc, and the corresponding correct outputs 1.ans, 2.ans, 3.ans, etc.

The default system verification method is to compare the sequence of tokens produced by the submitted program with those in the correct-output file. If they match, then the test case passes. This means that whitespace is ignored. If whitespace formatting or floating-point tolerances require checking, then the system validator can be told to do so (if you need any of these options, tell the problem-set editor).

It is crucial that the correct-answer files are indeed correct. There is nothing worse than telling contestants that their submitted program is not correct when it actually is. They lose points and waste time. Make sure your own program agrees with the correct-answer files and verify as many of the correct-answer files as possible via some other method (eg, by-hand, Wolfram Alpha, another person's program, etc).

The easiest output formats to verify are single integers. For example, instead of asking for the sequence of vertices that give the shortest path, instead ask for the weight of the shortest path. It's likely that knowing the weight implies knowing the path but is easier to verify.

Volunteer to judge. Occasionally, the problem you wrote has an unforeseen issue, such as an ambiguity in the writeup or error in the testcase data, that shows itself during a contest. The best person to fix the problem is the problem writer. Ideally, this means the problem writer is working as a judge and is readily available to patch things up.

Have fun and thank you. This is a creative activity, benefiting others, related to something you enjoy. It should be an enriching and satisfying thing to do. On behalf of all of the organizers and participating students, thank you for your help!
